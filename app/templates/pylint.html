Pylint

In Python

To all new programmers out there, here are some tips and tricks to understand the what and why of pylint and finally on how to implement pylint.

What is pylint?

Pylint is a tool that checks for errors in Python code, tries to enforce a coding standard and looks for code smells. It can also look for certain type errors, it can recommend suggestions about how particular blocks can be refactored and can offer you details about the code's complexity.

The Python community has formalized some recommended programming styles to help everyone write code in a common, agreed-upon style that makes the most sense for shared code. This style is captured in PEP 8, the "Style Guide for Python Code". Pylint can be a quick and easy way of seeing if your code has captured the essence of PEP 8 and is therefore friendly to other potential users.

Pylint will display a number of messages as it analyzes the code and it can also be used for displaying some statistics about the number of warnings and errors found in different files. The messages are classified under various categories such as errors and warnings.

Why should I use pylint?

Dynamic language like Python does not have a compiler that’ll catch production impacting bugs for you and plus helps you make your code more user readable and hence a linter like pylint would come to your rescue. Pylint has a lot of useful errors and warning, but also a whole lot of highly opinionated assumptions about how your code should look. And fundamentally it treats both opinions and objective problems the same way. There’s a distinction between warnings and errors, but both useful and useless stuff is in the warning category. Pylint can tell you where you may have run astray and point you in the direction to figure out what you have done and how to do better. You can run Pylint in a mode where it only complains about obvious errors—but then you’ll miss out on some of the important warnings that make Pylint so useful.

How to implement pylint

First step is to install pylint and there are 2 ways to do it.

  1. Python packages where it can be installed through pip
    • Command: pip install pylint (or it can be added in your requirements.txt that will be installed by your interpreter)
  2. Source distribution installation
    • By extracting the tarball: go to the extracted dir and run cmd: python setup.py install
    • In the editable mode using cmd: python setup.py develop
  3. For all the different IDEs there are specific ways to integrate pylint. Click here to know more.

Once you have installed it, it is a good idea to run pylint for a small section of your code to first understand what and how the output would look like. You can do that by calling pylint in the command line by either just hitting the cmd as pylint, which would traverse over all of your files and folders to check. Hence to make it easier you could comment all your lines of code except for a single file and run pylint.

Pylint report

Don’t be alarmed by the report you see at first as Pylint was created or in other words is only a way to mold you into a better programmer and to open your mind into the depths of programming. But be aware of the fact that pylint does not know any better than you, so be mindful of the changes that you would need to make or not make based on your project requirements.

Using the default text output, the message format is : MESSAGE_TYPE:LINE_NUM:[OBJECT:] MESSAGE

There are 5 kind of message types:

  • (C) convention, for programming standard violation
  • (R) refactor, for bad code smell
  • (W) warning, for python specific problems
  • (E) error, for probable bugs in the code
  • (F) fatal, if an error occurred which prevented pylint from doing further processing.

If you are using Pycharm, the pylint report would look something like the image on your right in the command line. It will tell you the error or warning is in which file of which folder, which line in that file and lastly the error or warning message. The report is pretty much straight forward. There are a list of error/warning messages, and you can look it up here.

Note: If you've considered what pylint has to say and decided that for this class, the code or logic you have is appropriate and reasonable, you can both suppress the error and indicate that you've considered the issue by adding a disabling comment to your class. For eg: pylint error: too-many-instance-attributes

How to change pylint report to your needs?

Pylint can be tailored to your needs by a way of either

  1. Through disabling or enabling the checks you would need for your project.
  2. By creating a custom-made configuration file that provides a list of checks for pylint along with custom-made error/warning messages to be shown. You can generate a config file via the cmd --generate-rcfile that will contain all the original checks and messages.
